## 0-1. ペインの抽出

「誰が・何に困っているか」の形式で列挙。

### 🧑‍💻 Web サイト訪問者のペイン
- **大規模サイト・ドキュメントサイトで自分が今どこにいるか分からない**
  パンくずリストでは「現在地」しか分からず、サイト全体の中での位置・周辺ページが見えない
- **全ページを俯瞰したいが、サイトマップページに飛ぶと文脈（読んでいたページ）が切れる**
  別タブで開く・戻るボタンで戻るなど、操作が冗長
- **階層が深いサイトでは関連ページへの移動コストが高い**
  3〜4階層のメニューを辿り直すのが面倒
- **モバイルから PC に切り替えたとき、PC ならではの広い画面を活かしたナビゲーションが無い**

### 🛠️ Web サイト運営者・開発者のペイン
- **訪問者の回遊率を上げたい / 直帰率を下げたい**
  現状ナビゲーションだけでは内部リンクの誘導が弱い
- **既存サイトに「導入1行」で済むナビゲーション補助が欲しい**
  CMS・SPA・SSG など環境がバラバラでも入れられるもの
- **サイト構造を変更してもプラグイン側を手動更新したくない**
  → sitemap.xml / クロールで自動追従するメンテフリー設計が必要
- **複数サイト（自社プロダクト群・ブログ群）に同じプラグインを使い回したい**
  → 汎用配布される npm パッケージである必要

### 🎨 デザイン視点のペイン
- **既存のサイトデザインを壊さずに重ねたい**
  色・フォント・余白がカスタマイズ可能で、テーマに馴染むこと
- **常時表示だとコンテンツの邪魔になる**
  → アイコン待機 → クリックで展開、というUXが必要

## 0-2. アプローチ3案の提示

要件「拡張性最優先 / npm 配布 / sitemap + クロール / 階層マップ / フローティング UI」を満たす実装方針を3案提示する。

### 🅰️ 案A：クライアントサイド完結型ライブラリ
**コンセプト：ブラウザだけで完結。サーバー不要。**

- ユーザーのブラウザ上で
  1. `/sitemap.xml`（および `sitemap_index.xml`）を fetch & パース
  2. パース結果に同一オリジンの内部リンクを DOM クロールで追加
  3. URL のパス階層からツリー構造を自動構築
- 結果は LocalStorage / IndexedDB にキャッシュ
- フローティングアイコン → クリックで階層マップウィンドウを展開

| 項目 | 内容 |
|------|------|
| ✅ メリット | 導入が `<SiteMap />` 1行で完結。バックエンド不要。CDN ホスティングも可能 |
| ⚠️ デメリット | 初回 fetch コスト。sitemap.xml が無いサイトはクロール頼みで遅い |
| 📈 拡張性 | クロール戦略・パーサーをプラガブルに（adapter パターン）すれば中規模まで対応可 |
| 🔧 技術相性 | React/Vue/Svelte などフロントエンド全般。SSR でも対応可（CSR fallback） |

---

### 🅱️ 案B：ビルド時静的生成型
**コンセプト：開発者がビルドステップでサイト構造 JSON を生成し、コンポーネントに渡す。**

- ビルド時に Next.js / Nuxt / Astro 等のルート情報や sitemap を読み取り、`site-tree.json` を生成
- ランタイムはその JSON を読むだけ → 超軽量・高速
- CLI ツール `site-map build` を同梱（フレームワーク別アダプタ）

| 項目 | 内容 |
|------|------|
| ✅ メリット | 表示が高速・確実。SEO/パフォーマンス◎ |
| ⚠️ デメリット | CMS 駆動サイト・動的サイトに弱い。フレームワーク非対応サイトは導入不可 |
| 📈 拡張性 | フレームワークごとにアダプタを書き続ける必要 → 拡張コストが線形に増える |
| 🔧 技術相性 | Next.js / Nuxt / Astro / Gatsby 等の SSG/SSR フレームワーク前提 |

---

### 🅲 案C：ハイブリッド型（クライアントクロール + Web Worker + 任意のキャッシュ API）
**コンセプト：案 A をベースに、重い処理を Web Worker に逃がし、オプションでサーバーキャッシュ層を載せる。**

- メイン処理は案 A と同じ（クライアントで sitemap + クロール）
- ただしクロールを **Web Worker** で実行 → UI スレッドを止めない
- IndexedDB に永続キャッシュ（TTL 制御）
- **オプション**：ホスト型 API（Cloudflare Workers 等）にクロール結果を保存 → 大規模サイトで初回も即時表示
- adapter パターンで「データソース」を差し替え可能（sitemap / API / 静的 JSON）

| 項目 | 内容 |
|------|------|
| ✅ メリット | 小規模〜大規模まで対応可能。UI 性能◎。拡張性が最も高い |
| ⚠️ デメリット | 構築コスト高。Worker / IndexedDB / オプションサーバーの実装が必要 |
| 📈 拡張性 | データソースを差し替えるだけで案 B 相当の挙動にもできる（最も汎用） |
| 🔧 技術相性 | 完全フレームワーク非依存（コアは vanilla TS、UI は React/Vue 別パッケージで提供） |

## ✅ 決定：案C（ハイブリッド型）採用

**選定理由：**
- 「複数サイト汎用配布／拡張性最優先」という非機能要件に最も適合
- フレームワーク非依存の vanilla core + 各 UI フレームワーク adapter という構造で、将来的に Svelte / SolidJS 等への展開も可能
- データソースを「sitemap fetch / DOM クロール / 静的 JSON / ホスト型 API」と差し替え可能にすることで、案A・案B の挙動も内包できる

**段階的リリース戦略：**
- **MVP（v0.1）**：案A 相当（クライアントクロール only、メインスレッド実行、LocalStorage キャッシュ）
- **v0.2**：Web Worker 化＋IndexedDB キャッシュ
- **v1.0**：オプションのホスト型 API（Cloudflare Workers）対応 → 大規模サイトで初回即時表示

## 0-3. ターゲットの明確化

```
ターゲットユーザー : Web サイト運営者・開発者
                    （ドキュメントサイト・コーポレートサイト・ブログ・社内ポータルなど、
                      ページ数が多く階層構造を持つサイトを運営している層）

二次ユーザー        : そのサイトの訪問者
                    （PC でサイト内を回遊しながら情報を探しているエンドユーザー）

ターゲットペイン    : ① 大規模サイトの訪問者が「自分が今どこにいるか／他にどんなページがあるか」を
                       同時に把握できない（パンくず＋サイトマップページの分離による文脈断絶）
                    ② サイト運営者が回遊率改善のためのナビゲーション補助を入れたいが、
                       既存サイトに低コストで導入できる汎用解が存在しない
                    ③ サイト構造変更のたびにナビゲーションを手動更新する保守負荷

解決アプローチ      : PC 画面左下に常駐するフローティングアイコンから、
                    クリック1回で「現在地ハイライト付きの全ページ階層ツリー」を表示。
                    sitemap.xml + 内部リンククロールにより**サイト変更に自動追従**。
                    npm パッケージ1行導入で運営者の保守コストをほぼゼロに。
```

### 想定ユースケース
| シナリオ | 効果 |
|---------|------|
| 大規模 OSS のドキュメントサイト | 訪問者が API リファレンス間を素早く回遊できる |
| 企業のサービス紹介サイト | 製品ラインナップ全体を直感的に把握できる |
| 個人ブログ（カテゴリ多数） | 過去記事の発見率が向上 → PV / 滞在時間改善 |
| 社内ナレッジポータル | 階層が深いドキュメントへのアクセスコスト削減 |

## 0-4. ツールの概念定義

> **`site-map` は、Web サイトに `<SiteMap />` を1行追加するだけで、画面左下に「現在地ハイライト付きの全ページ階層ツリー」をフローティング表示する npm パッケージです。**
>
> **sitemap.xml の自動取得と内部リンクのクローリングによりサイト構造を自動把握するため、運営者は導入後に一切のメンテナンスを行う必要がありません。Web Worker / IndexedDB により大規模サイトでも UX を損なわず、フレームワーク非依存の core + React/Vue/Svelte 等の adapter 構造で、将来の技術変化にも追従できます。**

### スケール見込み判断
- ✅ **スケールする**：複数サイトへの汎用配布が前提のため、拡張性を最優先で技術選定する
- ✅ **将来的なホスト型サービス展開可能性あり**：オプションサーバー（Cloudflare Workers）の上に SaaS 型ダッシュボード（クロール結果の可視化・分析）を載せる余地を残す
